Tu es Codex et tu dois continuer le développement de notre projet de hackathon Comutitres / Île-de-France Mobilités.

## 1. Contexte général du hackathon

Nous travaillons sur un hackathon autour de l’amélioration de l’expérience utilisateur Comutitres / Île-de-France Mobilités.

Le but n’est pas de refaire complètement le site IDF Mobilités ou de créer une application sans lien avec l’existant. Le but est d’améliorer l’expérience actuelle en apportant une vraie proposition de valeur métier, UX et technique.

Notre concept principal est :

> Compte Famille Navigo

L’idée est de permettre à une personne principale, appelée gestionnaire du foyer, de gérer plusieurs profils rattachés à son espace familial : enfant, jeune, étudiant, conjoint, personne âgée, proche aidé, etc.

Pour le MVP, nous développons surtout deux cibles fonctionnelles :

1. Jeune / enfant scolarisé

   * Exemple : enfant ou adolescent avec un forfait Imagine R / Navigo Junior.
   * Enjeux : renouvellement, justificatifs, photo, porteur/payeur, rentrée scolaire, anticipation des pics de charge.

2. Retraité / senior

   * Exemple : parent âgé ou retraité rattaché à l’espace familial.
   * Enjeux : accompagnement, lisibilité, aide à l’orientation vers Navigo Senior ou Améthyste, accessibilité.

Même si le MVP ne développe que ces deux cibles, l’interface doit montrer que le système est extensible à toutes les cibles familiales : conjoint, proche aidé, étudiant, bénéficiaire de réduction, etc.

Notre amélioration doit conserver cette logique rassurante et institutionnelle, mais en la modernisant et en ajoutant une vraie valeur :

> Passer d’un espace individuel “Mon espace / Mon Navigo” à un espace familial “Mon foyer Navigo”, où chaque profil dispose de son titre, de son statut et de ses prochaines actions.

## 4. Objectif de cette tâche

Créer un dashboard famille mobile-first, beau, clair et fonctionnel, qui devient l’écran central après l’inscription.

Ce dashboard doit être une amélioration du dashboard actuel IDF Mobilités.

L’objectif est que le gestionnaire du foyer puisse :

* voir tous les profils de sa famille ;
* comprendre rapidement le statut de chaque titre ;
* voir les actions prioritaires ;
* accéder au détail d’un profil ;
* lancer un renouvellement ou une recommandation de forfait ;
* voir les alertes importantes ;
* déclarer une perte de passe ;
* déclarer un passe trouvé ;
* comprendre que le système pourra gérer d’autres cibles plus tard.

Le dashboard doit donner une impression de produit complet, pas simplement une page de test.

## 7. Vision UX du dashboard

Le dashboard doit répondre à cette logique :

> Une personne principale gère plusieurs profils, chaque profil ayant son propre titre, son statut, ses actions et ses alertes.

Exemple :

Compte principal :

* Sophie Martin
* Gestionnaire du foyer
* Payeur principal

Profils :

* Sophie Martin — Moi — Navigo Annuel actif
* Lucas Martin — Enfant / jeune — Imagine R à renouveler
* Marie Dupont — Parent âgé / retraitée — Offre Senior à vérifier

Pour le MVP, on veut surtout afficher et rendre actionnables :

* le profil jeune ;
* le profil retraité.

Les autres profils peuvent être visibles comme possibilités d’ajout futures.

## 8. Adapter l’existant IDF Mobilités

Le dashboard actuel IDF Mobilités repose sur une structure :

* header institutionnel ;
* titre “Mon espace” ;
* navigation par onglets ;
* cartes d’action ;

Au lieu de :

Nous voulons :

> Mon foyer Navigo → Profils du foyer → Actions personnalisées par profil

Tu peux conserver une navigation inspirée des onglets actuels, mais adaptée au concept famille.

Exemple d’onglets possibles :

* Vue d’ensemble
* Profils
* Titres
* Services
* Alertes
* Aide

Ou bien une version mobile plus simple avec sections verticales.

Il faut éviter de faire un dashboard trop éloigné de l’identité IDFM. Le rendu doit rester crédible pour un service public.

## 12. Fonctionnalité perte / passe trouvé

Nous voulons intégrer l’idée suivante dans le dashboard :

* un membre peut déclarer la perte d’un passe Navigo ;
* une personne externe peut déclarer avoir trouvé un passe ;
* le propriétaire ne doit jamais être exposé publiquement ;
* le back-office pourra ensuite rapprocher le signalement.

Pour cette tâche dashboard, il faut au minimum :

### Déclarer une carte perdue

Depuis une action rapide ou une carte profil :

* afficher une modale de confirmation ;
* demander pour quel profil ;
* afficher un message de double confirmation :

  > Cette action peut bloquer le support actuel. Confirmez-vous la déclaration de perte ?
* créer un état mocké :

  > Demande de remplacement créée.

### Déclarer un passe trouvé

Créer une action rapide :

> J’ai trouvé un passe Navigo

Elle peut ouvrir une modale ou une page placeholder avec :

* numéro du passe ;
* lieu où il a été trouvé ;
* option : “Je l’ai déposé à un guichet”
* bouton : “Signaler le passe trouvé”

Ne jamais afficher l’identité du propriétaire.

Si le backend n’est pas prêt, stocker l’action en mock local proprement, mais prévoir une structure de service API.

Respecter :

* contrastes suffisants ;
* focus visible ;
* aria-label sur boutons iconiques ;
* modales accessibles ;
* boutons clairs ;
* texte non uniquement dépendant de la couleur ;
* labels explicites ;

Pour le profil senior, prévoir une interface particulièrement lisible :

* texte clair ;
* boutons faciles à comprendre ;
* explications courtes ;

## 20. RGPD / sécurité

Ne jamais afficher plus de données personnelles que nécessaire sur le dashboard.

Exemples :

* ne pas afficher de numéro de passe complet ;
* masquer les numéros sensibles ;
* ne pas afficher l’identité d’un propriétaire lors d’un passe trouvé ;
* ne pas exposer d’informations sensibles dans les mocks ;
* ne pas demander la civilité ;
* éviter les champs de genre ;
* garder une logique de minimisation.

## 23. Ce qu’il ne faut pas faire

Ne développe pas maintenant :

* vrai paiement ;
* vrai upload document complet ;
* vraie API FranceConnect ;
* vraie API CAF ;
* vraie API Améthyste ;
* vrai envoi d’email ;
* vrai SMS ;
* vraie logique complète de renouvellement ;
* vrai back-office complet ;
* gestion complète de toutes les cibles.

Ne code pas un énorme système générique qui risque de ne pas être fini.

On veut un MVP visible et cohérent.

## 26. Résultat attendu

À la fin, nous devons avoir un dashboard famille qui raconte immédiatement notre proposition de valeur :

> Une personne peut gérer les titres de transport de toute sa famille depuis un seul espace, avec des profils adaptés selon l’âge et la situation : jeune à renouveler avant la rentrée, retraité à accompagner vers une offre senior, et à terme conjoint, proche aidé, étudiant ou bénéficiaire d’une réduction.
